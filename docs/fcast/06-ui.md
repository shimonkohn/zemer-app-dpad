# 06 — The cast UI

All cast UI is Material 3 *standard*, fully D-pad navigable (shared
`focusBorder`), uses theme tokens (no hardcoded colours/sizes), and localizes
every string into the default `app/src/main/res/values/metrolist_strings.xml`.
`scripts/ui-audit.sh` enforces no new hardcoded strings / hex / font sizes / raw
`AlertDialog`s.

## Entry points

| Surface | File | Look |
| --- | --- | --- |
| Full-player artwork | `ui/player/Thumbnail.kt` → `CastButton` | dark-disc icon overlaid top-right of the artwork |
| Mini player (new) | `ui/player/MiniPlayer.kt` | bordered circular chip (primary tint when connected) |
| Mini player (legacy) | `ui/player/MiniPlayer.kt` | plain `IconButton` |
| Settings | `ui/screens/settings/PlayerSettings.kt` | "Enable casting" `SwitchPreference` |

All four are gated on `castEnabled || connectedDevice != null` (visibility), and
all open the picker through the **one** shared launch helper.

## The shared launch helper

```kotlin
// CastButton.kt — the single launch path + shared state/icon, used by CastButton + both mini-player buttons
fun openCastPicker(playerConnection: PlayerConnection, menuState: MenuState) {
    playerConnection.service.startDiscovery()
    menuState.show {
        CastPicker(playerConnection, playerConnection.mediaMetadata.value) { menuState.dismiss() }
    }
}

@Composable fun rememberCastButtonState(): CastButtonState?  // null when hidden; collects the flow + pref once
@Composable fun CastIcon(connected: Boolean, idleTint: Color, size: Dp)  // the cast/cast_connected glyph
```

The three cast buttons keep their **distinct wrappers** (the artwork's dark disc,
the new mini-player's bordered circle, the legacy mini-player's plain
`IconButton`), but the shared parts are no longer duplicated: each calls
`rememberCastButtonState()?.let { … CastIcon(it.connected, …) }`, so the
visibility gate, the connected-flow + `CastEnabledKey` collection, and the icon
live in one place. Change the launch behaviour or the gate once, here.

## The picker (`CastPicker`)

Shown via the shared menu bottom-sheet host (`LocalMenuState`). It is
self-contained: it collects its own state (`connectedDeviceFlow`,
`discoveredDevicesFlow`, `castLibState`) and resolves the stream URL at connect
time. It renders one of these states:

- **Lib not present** (`Idle`/`Downloading`/`Failed`) — consent text + Download,
  a progress bar, or an error + Retry. Mirrors `CastDownloadDialog` so the two
  are consistent.
- **Ready, connected** — a "Stop casting" row for the connected device.
- **Ready, searching** — spinner + "Searching for cast devices…" + "same
  Wi-Fi" hint + the FCast-install section (Chromecast and FCast are both named
  by the settings toggle and the install link's "No Chromecast handy?" copy).
- **Ready, devices found** — a `CastDeviceRow` per device; tapping connects. The
  tapped row swaps its icon for a spinner + "Connecting…" subtitle, and further
  taps are ignored until the attempt resolves.

While Ready and not connected, the header carries a **refresh button** (spinner
while a refresh is in flight, state from `castDeviceRefresher.refreshing`), and
opening the picker triggers one automatic `castDeviceRefresher.refresh()` so the
list reflects what is advertised *now* — see the refresh-burst section in
[03](03-discovery-and-connection.md).

`connect(device)` delegates to `service.castConnector.connect(...)`
(`playback/CastConnector.kt`) on **`service.scope`** — the service-lifetime
scope, *not* the Activity-bound `playerConnection.scope`: backgrounding the app
mid-connect disposes the latter, which would cancel the coroutine, skip the
connector's timeout abort (`handler.disconnect()` on `TIMED_OUT` — the pending
SDK attempt could then surprise-connect later), and strand the row spinner
(`connectingDevice` is reset in a `finally`, so every outcome — including
cancellation — re-enables the rows). The picker only maps the result to UI:
`Connected` dismisses the sheet, `NoStream` / `Failed` show a toast. The
orchestration itself (pause local → resolve stream URL → click-time NSD address
re-resolve → `handler.connectTo(...)` → `markRemoteLoaded` → await the
Connected/Disconnected outcome) lives in the connector — see
[03](03-discovery-and-connection.md). The sheet is **not** dismissed on tap: a
failure must be visible, not silent.

## The consent dialog (`CastDownloadDialog`)

Shown the moment casting is enabled in Settings (the lib isn't bundled). Built on
the shared `DefaultDialog` (not a raw `AlertDialog`). Same states/strings as the
picker; auto-dismisses once `Ready`, offers Retry on failure.

Both surfaces render the download as a **determinate progress bar** + a
"Downloading cast support… N%" label driven by
`CastLibState.Downloading.progress` (the 0..1 fraction), falling back to an
indeterminate bar while the fraction is `null` (no usable `Content-Length` yet —
see [02](02-on-demand-native-lib.md)). A one-shot **"Cast support downloaded"
toast** fires on the `Downloading → Ready` transition, and only on the surface
that initiated the download (an already-`Ready` recomposition must not re-toast).

## Shared components & conventions used

- **Dialog:** `DefaultDialog` (`ui/component`).
- **Settings rows:** `PreferenceGroupTitle` + `SwitchPreference`.
- **Focus/D-pad:** `focusBorder(...)` on every clickable element.
- **Type/colour:** `MaterialTheme.typography.*` / `MaterialTheme.colorScheme.*`
  throughout. The only raw colour is the artwork cast button — a solid dark disc
  (`Color.Black.copy(alpha = 0.45f)`) with a white icon, the standard
  legible-over-arbitrary-artwork exception. (A radial black *scrim* was tried first
  but vanished on dark album art — black on dark — hiding the button.)

### Intentionally bespoke

- `CastCircleButton` (the share/copy round buttons) and `CastDeviceRow` (picker
  rows) are small purpose-built composables — built on the shared primitives, but
  no shared component matches their shape, so they aren't "hardcoded," just local.

### Open alignment items (not yet done — would change pixels / add cross-cutting infra)

- The copy-link button uses a dedicated `content_copy.xml`, whereas the rest of
  the app uses `R.drawable.link` for copy-link affordances.
- `shareFcast` / `copyFcast` re-implement the share-intent and clipboard-copy
  patterns; there is **no** shared `shareText`/`copyToClipboard` helper in the app
  yet (the same boilerplate exists in ~10 menus). Extracting one is the clean fix
  but touches many non-cast files.
